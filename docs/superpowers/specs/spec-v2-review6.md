
## Real issues

**`result <- chk.Run(...)` — `result` is undeclared.** The goroutine example now correctly calls `chk.Run`, which is the fix from last round. But it sends to `result` which is never declared in the snippet. An implementer copying this sees an undefined variable. Either declare it above the loop — `results := make(chan runResult, len(activeCheckers))` — or change the comment to describe the channel pattern without showing the send syntax. The example is close to correct; one line of declaration finishes it.

**The `latest` resolution policy has a logic gap when the tool IS installed but the `go list` call fails.** The spec says: if tool is not on PATH, skip `go list` and install directly. If `go list` fails for network reasons, fall back to installed version with `unknown` in the cache. But it doesn't cover the case where the tool IS on PATH, `go list` SUCCEEDS and returns a newer version, but the *comparison* shows the installed version is older — triggering a re-install. That re-install then calls `go install <pkg>@latest`, which might fail if the network disappears between the `go list` call and the `go install` call. What's returned to the agent — a Diagnostic error? A partially warm cache? This is the classic TOCTOU window in version-checking. Worth one sentence: "if re-install fails after a successful `go list`, the server falls back to the already-installed version, logs the failure to stderr, and caches the installed version as `unknown` to prevent re-install loops on subsequent requests."

**`osv.details` as a fallback field is not part of the govulncheck NDJSON output.** The extraction rule says: "If `osv.summary` is absent, use `osv.details` truncated to 120 characters." The govulncheck NDJSON emits `osv` objects that follow the OSV schema. In the OSV schema the long-form description field is called `details` at the top level. However, the govulncheck handler reads the `osv` lines from its NDJSON stream — and what govulncheck actually emits in its `osv` line is a subset of the full OSV record. Govulncheck's NDJSON `osv` object typically includes `id`, `summary`, `aliases`, and `affected` but not always `details` — the `details` field is often omitted in the streamed output even when present in the full OSV database record. Depending on which version of govulncheck is installed, relying on `osv.details` as a fallback may silently produce empty strings. The safer fallback is the `finding.osv` ID, which is always present. Consider dropping `osv.details` from the fallback chain and going straight to the ID, or noting this risk explicitly.

## Minor issues

**Auto-Install Rule 1 says "discover each tool via `exec.LookPath`" but the pre-flight is scoped to only the tools about to run.** The rule says "on the first incoming request... discover *each* tool" — implying all three are discovered. But the Pre-Flight Sequential Install section correctly says only the tools about to run are discovered and installed. These two descriptions are inconsistent. Rule 1 should say "discover each *requested* tool via `exec.LookPath`."

**The goroutine closure captures `check` by reference.** The loop variable `check` is used as the goroutine argument `chk Checker` — which is the correct pattern (the argument is passed by value, avoiding the closure capture bug). But the Data Flow pseudocode on line 144–149 uses a closure `go func() { ... }()` with no parameter, which *does* capture the loop variable by reference. The pseudocode and the Timeout Model example are inconsistent about this. The Timeout Model pattern is correct; the Data Flow pseudocode has the classic Go loop-closure bug baked in.

**The `result <-` channel send in the Timeout Model example sends the return value of `chk.Run` directly, but `chk.Run` returns `([]Diagnostic, error)` — two values.** A channel send `result <- chk.Run(...)` doesn't compile when the function returns two values. The channel type would need to be a struct wrapping both, or the two return values need to be captured separately before sending. Again, this is fixable with one or two lines of scaffolding in the snippet.

**`install_tools` output format shows `version: "latest"` in the `failed` entry.** The spec established that the cache stores resolved versions, not the requested specifier. But if the install failed, there was no successful resolution — so `latest` is technically correct here (the requested version, not a resolved one). It's fine, but worth a note on the `failed` entry description: "`version`: the requested specifier if resolution failed before install, or the resolved version if install itself failed."
